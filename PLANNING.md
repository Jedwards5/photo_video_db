# Personal Memories Archive — Project Planning

## Project Overview

A local-first, privacy-preserving personal media archive that ingests Snapchat memory exports, enriches them with AI-generated metadata (face recognition, semantic tagging, reverse geocoding), and stores everything in a queryable SQLite database. Designed to be extensible to other photo sources (iPhone, etc.) in the future.

---

## Core Design Principles

- **Local-first, no cloud.** All processing and storage happens on the user's machine. No images or metadata are ever sent to external APIs.
- **Originals are sacred.** Source files are never modified. All enrichment lives in a sidecar SQLite database.
- **Source-agnostic architecture.** Snapchat is the first data source but the system is designed from day one to accommodate others (iPhone Photos, Instagram exports, etc.).
- **CLI first.** Start with a command-line interface. A local web UI (for mobile access via home network or Tailscale) is a future evolution.

## SQLite Schema Considerations

Every record should carry a `source` field so queries can be source-agnostic or filtered by origin. Suggested tables:

- `media` — one row per photo/video, with filepath, timestamp, source, media type
- `people` — named face clusters
- `face_appearances` — join table linking media to people
- `tags` — semantic/scene tags generated by CLIP
- `locations` — reverse-geocoded place names linked to media

---

## Enrichment Pipeline (build in this order)

1. **Ingest & normalize** — parse `memories_history.json`, populate `media` table with existing metadata (timestamps, coordinates if present, media type)
2. **Reverse geocoding** — convert any GPS coordinates to human-readable place names using `geopy` + Nominatim (OpenStreetMap, free, network requests go to OSM only)
3. **Face detection & clustering** — detect faces, generate embeddings, cluster similar faces, allow user to name clusters; stored in `people` + `face_appearances`
4. **Semantic tagging via CLIP** — run `open-clip` or HuggingFace `transformers` locally to tag scenes and objects; enables natural language search (e.g. "birthday cake")
5. **Video transcription** — run `openai-whisper` locally on MP4s to generate searchable text from audio
6. **Query interface** — build CLI query layer last, once you know what data you actually have

## Local-Only Model Stack

All of the following run entirely offline with no API keys required:

| Purpose | Library | Notes |
|---|---|---|
| Semantic image search | `open-clip` or HuggingFace `transformers` (CLIP) | Open-source, released by OpenAI, runs locally |
| Face detection & clustering | `face_recognition` | Built on dlib |
| Reverse geocoding | `geopy` + Nominatim | Network requests to OSM only |
| Video transcription | `openai-whisper` | Open-source, runs locally |
| Metadata store | `sqlite3` (stdlib) | Single local file |

> **Note on naming:** CLIP and Whisper are open-source models originally created by OpenAI researchers, but using them locally requires no OpenAI account, API key, or data transfer to any server. They are not the same as the OpenAI API.

## Source Adapter Pattern (`base.py`)

Each data source (Snapchat, iPhone, etc.) should implement a common interface so the ingest pipeline doesn't care about source-specific logic. At minimum:

- `parse_metadata()` — reads source-specific export format, returns normalized records
- `get_media_files()` — returns list of media file paths
- `normalize_record()` — maps source fields to the unified schema

Adding a new source in the future means writing one new adapter file without touching enrichment or query logic.

## Future: iPhone Photos

iPhone EXIF data is richer than Snapchat's export in some ways (native GPS, device info, precise timestamps). Key considerations when that time comes:

- Export via the Mac Photos app preserves EXIF fully
- `osxphotos` library can programmatically export from the Mac Photos library including Apple's existing face clusters and album metadata — potentially skipping face recognition for that source entirely
- User does not currently have a Mac, so iPhone ingest approach is TBD

## Web Interface

### Stack
- **Backend:** FastAPI + Uvicorn (Python)
- **Frontend:** Vue 3 + Vite
- **Auth:** Single password (`APP_PASSWORD` in `.env`) → JWT token, 7-day expiry, stored in browser localStorage
- **Design:** Mobile-first, 2-col grid on phone / 3-4 col on desktop

### Views
| View | Description |
|------|-------------|
| Login | Password form, stores JWT on success |
| Timeline | Infinite scroll grid grouped by month/year |
| People | Named face clusters → click into person gallery |
| Search | Natural language (CLIP), tag, and transcript search |
| Detail | Full media + metadata panel (date, location, tags, people) + prev/next nav |

### Build Phases

**Phase 0 — Prerequisites**
- Install Python deps: `fastapi uvicorn python-jose[cryptography] python-multipart python-dotenv`
- Install Node.js
- Scaffold frontend: `npm create vite@latest web -- --template vue`
- Create `.env` with `APP_PASSWORD` and `SECRET_KEY` (gitignored)
- Commit `.env.example` showing required variable names without values

**Phase 1 — Thumbnail generation** (`src/thumbnails.py`)
- Run once before launching the server: `python -m src.thumbnails`
- Photos: Pillow resize to 400px JPEG → `data/processed/thumbnails/{id}.jpg`
- Videos: ffmpeg extract first frame → same path
- Idempotent — skips already-generated thumbnails

**Phase 2 — FastAPI backend** (`src/api/`)
- `main.py` — app init, CORS, serves Vue `dist/` as static files in production
- `auth.py` — password verify → issue JWT; middleware protects all `/api` routes
- `routes/media.py` — `GET /api/timeline` (paginated), `GET /api/media/{id}`, `GET /files/{id}` (stream original), `GET /thumbnails/{id}`
- `routes/people.py` — `GET /api/people`, `GET /api/people/{id}/media`
- `routes/search.py` — `GET /api/search?q=` (CLIP), tag search, transcript search

**Phase 3 — Vue frontend** (`web/src/`)
- Vue Router for navigation between views
- `api.js` — shared fetch wrapper that attaches JWT header, redirects to Login on 401
- Video items display play button overlay on thumbnail in grid
- Detail view shows video player (HTML5 `<video>`) or full-res photo + metadata sidebar

**Phase 4 — Integration**
- Dev: Vite proxy (`/api → localhost:8000`) so both servers run independently
- Prod: `npm run build` → FastAPI serves `web/dist/` at `/`
- Single prod command: `uvicorn src.api.main:app --host 0.0.0.0 --port 8000`

### Access
- **Now:** `http://localhost:8000` or `http://192.168.x.x:8000` on home network
- **Future:** Tailscale VPN for remote access — no code changes needed, just install Tailscale on the host machine
